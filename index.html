Artifact One - Software Design & Engineering

GitHub Repository: The Loop

This project began as a simple event-tracking CRUD app for the class CS-360 Mobile Architecture. While it fulfilled the requirements for that class, there were several issues preventing it from being a fully realized app: First of all, the user login details and database of events were SQLite databases, with login details stored in plaintext. There were also several issues with the separation of concerns, where the UI layer could interact directly with the database. Finally, there was stub code for a SMS notification system that was incorrectly implemented. While there was significant overlap with the third enhancement (databases), I was careful to keep the scope of this enhancement solely focused on the app’s architecture.

Enhancements:
  • Replaced a flawed plaintext SQLite login system with a more secure Firebase Authentication system
  • Architectural improvements like replacing direct database calls in the UI layer with a repository for better separation of concerns
  • Replaced incomplete/deprecated SMS notification code with local push notifications

Why I selected it. I chose The Loop because I built it from scratch. It's relevant to my interests, not from a template, and it's relevant to an industry I care about. More importantly, it had clear, concrete weaknesses I wanted to fix. The plaintext credential storage was weighing on me - storing passwords in a SQLite database with no hashing is a security vulnerability that would be unacceptable in any real deployment. Beyond security, the app had architectural issues that I didn't fully realize as a novice developer earlier on in the program: UI activities were interacting directly with the database instead of going through a data layer, making the code harder to test and maintain. I replaced the SQLite login with Firebase Authentication, which handles password hashing, rate limiting, and session management without the app storing a credential itself. I also created a new EventRepository class as the single point of access to the database, so that every activity and adapter talks to the repository rather than opening its own database connection. Finally, I removed the deprecated SMS notification stub and replaced it with working push notifications. My skills demonstrated here include secure authentication, software architecture, and Android development.

Reflection. My original enhancement plan described a manual SHA-256 hash-and-salt implementation, and I had to revise that plan after research. SHA-256 is not recommended for modern password hashing. It is too fast, making it easier to brute-force, and Firebase's built-in authentication uses bcrypt along with rate limiting and session expiration. Choosing Firebase over my original plan was a demonstration of a security mindset: setting aside my own design in favor of what was actually best for the application and its users. The other challenge was managing scope between this enhancement and the databases enhancement. There is significant overlap between tthe two enhancements and I was careful to keep this module focused specifically on authentication and separation of concernsrather than the data model itself.

Artifact Two - Algorithms & Data Structures

GitHub Repository: FragranceDNA

FragranceDNA solves a practical problem: people new to perfumery often struggle to articulate what they like beyond naming specific fragrances that they enjoy. This project addresses that in two ways – given a fragrance that the user enjoys, it returns a ranked list of similar fragrances, and it identifies which categories best describe their taste, giving users a vocabulary to use when exploring the world of fragrance.

This project was based on coursework from DAT-325 (Data Validation: Quality and Cleaning) and MAT-350 (Applied Linear Algebra), and was influenced by my own independent research, particularly Aggarwal’s Recommender Systems: The Textbook, from which I developed the approach of TF-IDF weighting and cosine similarity. The nonnegative matrix factorization (NMF) aspect brings the project into machine learning territory, where the algorithm independently discovers 15 distinct fragrance ‘archetypes’, which I then interpreted and named based on my own knowledge of perfumery.

Enhancements:
  • Cleaned the dataset scraped from Fragrantica.com, addressing capitalization issues and dashes and fixing 40 note variants (misspellings, synonyms, and regional variants)
  • Improved matrix construction from an O(n) linear search to O(1) dictionary lookup, achieving a 25x speedup in creating the original fragrance x note matrix
  • Replaced a basic Jaccard similarity algorithm with TF-IDF weighting and cosine similarity, better handling fragrances with different numbers of notes
  • Implemented non-negative matrix factorization at k=15 to discover fragrance archetypes
  • Deployed the entire system as an interactable, browser-based recommender using Pyodide (Python via WebAssembly), without the need for a backend server

Why I selected it. I selected FragranceDNA as a personal project containing the skills I'm most proud of: applied mathematics, data quality work, and my strong research skills as a librarian. For the data layer, I manually reviewed over 1,600 unique fragrance notes in Excel, correcting 40 note variants including misspellings, synonyms, and regional alternatives before writing any cleaning code. I replaced an O(n) linear search through a list of notes with an O(1) dictionary hash map lookup, achieving a 25x speedup in matrix construction. I also replaced a basic Jaccard similarity metric with TF-IDF weighting and cosine similarity, which handles fragrances with different numbers of notes more fairly. Finally, I applied non-negative matrix factorization at k=15 to let the algorithm discover 15 distinct fragrance archetypes independently, which I then looked through and named using my own knowledge of perfumery. Deploying the entire system as a browser-based demo using Pyodide required a lot of additional research and trade offs — the NMF decomposition was too memory-intensive to run in-browser, so I precomputed the W matrix and saved it as JSON, while cosine similarity and TF-IDF weighting run in the browser.

Reflection. Implementing the recommender as a browser-based demo was the biggest challenge of this enhancement. Before the capstone, I had a very clear vision of the algorithm but no idea how to make it usable by other people beyond sending them a csv file and a Jupyter notebook. Pyodide, which runs Python via WebAssembly, solved the backend problem, but getting sklearn's NMF to run in-browser proved to be too memory intense even for a decent gaming computer. Precomputing the NMF decomposition was the trade-off I decided on, even though it meant the browser demo doesn't run that step live. I also made two changes to the algorithm itself after testing: I excluded vague "X notes" descriptors (like "woody notes," "floral notes") from the NMF step because they were grouping fragrances by the vague descriptions rather than by actual similarity, and I raised k from 4 to 15 to produce better archetypes for a beginner audience.

Artifact Three - Databases

GitHub Repository: The Loop

Revisiting The Loop for the final enhancement, I made significant improvements to the database. As my choice for migrating from SQLite, I chose Firebase Firestore to round out my database experience. Most of my professional work is with Postgres, and I was curious about the document model because it has overlap with how metadata records are structured and because it suits nascent projects that don't yet need a strict relational schema.

Enhancements:
  • Migrated the database backend from two separate SQLite databases to a unified Firebase Firestore database (a NoSQL document model)
  • Redesigned the data model to include a user-event association to enable features like event reminders and a history tab
  • Seeded the database with real event data from Ticketmaster's Discovery API

Why I selected it. I chose Firestore deliberately rather than defaulting to a relational database because most of my professional work is already in PostgreSQL, and I wanted to build real experience with a document model. I found that the document model suits early-stage app development well which matters when you are quickly iterating on an app's design and data needs. The Ticketmaster integration demonstrates use of a real production API. Though I was limited with the free tier, I successfully seeded the database with thousands of events. The skills demonstrated here include document data modeling, schema design, third-party API integration, and pagination.

Reflection. The biggest challenge was the gap between what the original CS-360 app needed and what a real multi-user app requires. The original was designed around a few hard coded events. Scaling to thousands of events meant rethinking almost everything. How many events should load at once? Should the list paginate or infinite-scroll and how does that affect app performance? What happens to events once they've passed? Where do users find the events they've starred? Can users submit their own events, and if so, how is that moderated? These questions came up in the planning stage of this enhancement, and answering them required a bit of restructuring. The "history" tab became "my events" showing upcoming starred events rather than past ones because upcoming events are more useful to a user. I also had to lay the groundwork for the first enhancement (authentication and the EventRepository layer) before this enhancement could be built on top of it, which made the sequence of these two enhancements importnat to think through.
