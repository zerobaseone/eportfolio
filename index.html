Professional Self-Assessment

Throughout the CS499 Capstone, I've had the opportunity to reflect on my growth as a computer scientist — synthesizing skills developed across the program into three meaningful enhancements that demonstrate both breadth and depth.

I've been drawn to code since the early 2000s, when the web was something you could take apart and put back together — customizing HTML pages and tinkering with Java applets taught me early that software was less magic and more craft. I studied computer science formally in college in 2012, and while several career moves followed, I eventually found myself in libraries. I stayed for ten years.

Those years were anything but a detour. I built deep expertise in Excel, VBA, and databases — tools I used daily to automate workflows, untangle data problems, and make processes faster and less painful for the people around me. Finding out that something I built quietly made someone's job easier became its own reward, and it shaped how I think about what software is actually for.

Now I've returned to computer science with fresh perspective and renewed purpose. What started as curiosity about early web customization has grown into a serious interest in big data and data science. This program — and this capstone — represents the convergence of that early spark, a decade of applied experience, and the formal foundations I've been building ever since.

My background as an artist and librarian has given me an unusual lens: I approach data the way I'd approach a collection — with purpose, organization, and the end user always in mind. The three artifacts in this portfolio represent the convergence of those instincts with formal computing knowledge.

My librarianship informs every stage of my work in software design, data analysis, and data cleaning. Libraries are built on the principle that information should be findable and usable by anyone who needs it — and I carry that into my code. Years of reference work taught me that what someone asks for and what they actually need are not always the same thing, and that good design anticipates the gap.

Working across the library with patrons of widely varying abilities and backgrounds has made accessibility a core value rather than an afterthought. But I try to extend that thinking beyond end users: software is also read by collaborators, reviewed by peers, and inherited by developers who weren't part of the original conversation. I write with all of them in mind.

Artifact One - Software Design & Engineering

The first artifact is a mobile application called "The Loop" that allows users to find live music events in Chicago's local music scene. This application was originally created for CS-360 (Mobile Application Development) using a combination of Java and XML in Android Studio.

Narrative: The development process thus far was largely focused on building the CRUD components of the application, allowing users to create, update, and delete events. The login process, user interface, and database are still very simplistic, but the application is functional and allows users to interact with the data. In this enhancement, I will be adding a more robust login process and make adjustments to the user interface, focusing on accessibility and aesthetics.

Improvements
  • Develop a more secure login process
  • Test with Postman
  • Ensure all links work
  • Make the user interface more accessible and aesthetically pleasing
